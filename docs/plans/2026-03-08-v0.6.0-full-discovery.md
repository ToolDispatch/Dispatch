# Dispatch v0.6.0 — Full Discovery: All Sources, All Tools

**Goal:** Expand Dispatch from ranking *installed* tools to discovering and ranking *all available* tools — installed or not — across every queryable source. Haiku scores the full ecosystem against the task, not just what the user already has.

**Problem being solved:**
Currently Dispatch only feeds installed plugins to Haiku for ranking. Marketplace catalogs contain descriptions for every plugin (installed or not) — we're ignoring uninstalled ones. The official MCP registry is publicly queryable with no auth and returns descriptions — we're not calling it. The user has no way of knowing what tools exist for their task. That is the primary value Dispatch should deliver.

---

## Two-Phase Architecture

### v0.6.0 — Local catalog + cron (this plan)

A background cron job builds and maintains a local `tool_catalog.json` by aggregating all sources. The hook reads this file at runtime — zero network calls in the critical path except the Haiku/Sonnet ranking call.

**Why cron instead of fetching at hook time:**
- Hook budget is 10s total. Network calls to 3+ sources at hook time is unreliable.
- A stale-by-24h catalog beats a timeout that kills the hook.
- Cron can paginate deeply, retry failures, and deduplicate across sources without time pressure.

### v0.7.0 — Server-side catalog + embeddings (next plan)

The Dispatch server maintains the aggregated catalog and pre-computes embeddings for every tool description. When `/rank` is called:
1. Server does semantic similarity search (embeddings) against the task + context → top-N candidates
2. Sonnet re-ranks just those candidates with full reasoning
3. Free users get Haiku ranking over local catalog (no embeddings)
4. Pro users get embedding pre-filter → Sonnet final ranking over the best candidates

**Why this matters:** Embeddings find semantically relevant tools with zero keyword overlap. Sonnet reasoning over 20 pre-filtered candidates beats Haiku reasoning over 200 raw descriptions. The server can also track which recommended tools users actually activate — a real quality signal no registry has.

## Verified Sources (no guessing)

### 1. Local marketplace catalogs — already present, unused

Every registered marketplace downloads its full catalog to:
```
~/.claude/plugins/marketplaces/{name}/.claude-plugin/marketplace.json
```

Schema (confirmed from cc-marketplace, claude-code-plugins, claude-plugins-official):
```json
{
  "plugins": [
    {
      "name": "sugar",
      "description": "Transform Claude Code into an autonomous AI development powerhouse...",
      "keywords": ["autonomous", "workflow", "productivity", "agents"],
      "category": "workflow",
      "source": "./plugins/sugar"
    }
  ]
}
```

Currently registered marketplaces on this machine:
- `cc-marketplace` — 119 plugins
- `claude-code-plugins` — 13 plugins
- `claude-plugins-official` — 56 plugins
- `github.com-VoltAgent`, `github.com-travisvn` — user-added

**We read plugin.json (installed only). We should read marketplace.json (all) and filter by task keywords.**

### 2. Official MCP Registry — public, no auth

- **Endpoint:** `GET https://registry.modelcontextprotocol.io/v0.1/servers?search=<term>`
- **Auth:** None required for reads
- **Response:** `{servers: [{server: {name, description, repository: {url}}}]}`
- **Verified with:** `curl "https://registry.modelcontextprotocol.io/v0.1/servers?search=github&limit=3"`
- **Source:** https://github.com/modelcontextprotocol/registry
- **Note:** Search is substring-only. Use primary task term. Cache 1hr.

### 3. skills.sh — already used, confirmed

- **Endpoint:** `GET https://skills.sh/api/search?q=<term>&limit=10`
- **Auth:** None
- **Source:** https://github.com/vercel-labs/skills/blob/main/src/find.ts
- Currently called via `npx skills find` — could call directly if npx becomes unavailable

### 4. Smithery.ai — MCP servers, requires auth

- **Endpoint:** `GET https://registry.smithery.ai/servers?q=<term>`
- **Auth:** Bearer token required — user must provide Smithery API key
- **Not included in v0.6.0** — optional, opt-in only (add token to config.json)

## Architecture Change

### Current flow (evaluator.py)

```
scan_installed_plugins()     → installed plugins only
scan_mcp_servers()           → installed MCP servers only
get_installed_skills()       → installed skills only
search_registry(task_type)   → uninstalled agent skills only
                               ↓
rank_recommendations(installed plugins, installed MCPs, installed skills, uninstalled skills)
```

### v0.6.0 flow

```
load_tool_catalog()           → ALL tools from local catalog (installed + uninstalled, all sources)
scan_mcp_servers()            → installed MCP servers — mark matching catalog entries as installed
get_installed_skills()        → installed skills — mark matching catalog entries as installed
                                        ↓
rank_recommendations(
  all catalog tools,    # full ecosystem, no pre-filter — LLM does semantic matching
  task_type,
  context_snippet
)
                                        ↓
score gap truncation          → cut list at first ≥25-point gap — only solid matches surface
```

**No keyword pre-filtering — the LLM is the filter.** Send all tools to Haiku/Sonnet. 200 plugins × ~150 chars ≈ 28K input tokens, well within the 200K context window. Semantic matching by the LLM is the point — a Docker task might need a tool whose description says "containerized deployment environments" with no keyword overlap. Keyword filtering would break that.

**Score gap truncation is the quality gate.** After ranking, the existing 25-point cliff cut removes everything below the first significant score drop. Irrelevant tools get low scores, land below the cliff, and never appear. This is especially important with a large pool — the gap filter ensures only solid matches surface.

## Implementation Plan

### Task 1: `build_tool_catalog()` + cron refresh — replaces `scan_installed_plugins`

**Files:** `evaluator.py`, new `catalog_refresh.py`, cron entry

**What:** Maintain a local `~/.claude/skill-router/tool_catalog.json` built from all sources. The hook reads this file — zero network calls in the critical path.

**Catalog schema per entry:**
```json
{
  "id": "marketplace/cc-marketplace/sugar",
  "name": "sugar",
  "description": "Transform Claude Code into an autonomous AI development powerhouse...",
  "keywords": ["autonomous", "workflow", "agents"],
  "category": "workflow",
  "source": "marketplace",
  "marketplace": "cc-marketplace",
  "install_cmd": "/plugin install cc-marketplace/sugar",
  "url": "https://github.com/cdnsteve/sugar",
  "installed": false
}
```

**`catalog_refresh.py`** — standalone script, called by cron daily:
1. Read every `~/.claude/plugins/marketplaces/*/.claude-plugin/marketplace.json` → all plugins with descriptions
2. Fetch `https://registry.modelcontextprotocol.io/v0.1/servers?search=<common_terms>&limit=50` → MCP servers (paginate if needed)
3. Fetch known public marketplace catalogs for users who haven't registered them (anthropics/claude-code, ananddtyagi/cc-marketplace, anthropics/claude-plugins-official raw GitHub URLs)
4. Cross-reference installed state: `scan_mcp_servers()` + `get_installed_skills()` + plugin dir scan
5. Merge, deduplicate by id, write `tool_catalog.json`

**`load_tool_catalog()`** in `evaluator.py` — reads the catalog file, marks current installed state, returns full list. Falls back to current `scan_installed_plugins` behavior if catalog doesn't exist yet.

**Returns:**
```python
[
  {"name": "sugar", "description": "...", "marketplace": "cc-marketplace", "installed": True, "source": "marketplace"},
  {"name": "n8n-workflow-builder", "description": "...", "marketplace": "cc-marketplace", "installed": False, "source": "marketplace"},
]
```

**Plugin install_cmd — VERIFIED:** `claude plugin install {name}@{marketplace}` is the correct terminal command. Confirmed via `claude plugin --help`. Install command per catalog entry: `f"claude plugin install {name}@{marketplace} && claude"` (chains relaunch).

### Task 2: `search_mcp_registry(task_type)` — new function

**File:** `evaluator.py`

**What:** Call `https://registry.modelcontextprotocol.io/v0.1/servers?search=<term>&limit=5` for each unique term in the compound task type (same logic as `search_registry` — up to 3 terms, deduplicated by server name). Cache results 1hr in `npx_cache.json` under key `"mcp_registry"`.

**Cross-reference with installed:** After fetching, mark any server as `installed: True` if its name or repository URL matches a known installed MCP (from `scan_mcp_servers()`).

**Returns:**
```python
[
  {"name": "io.github.owner/mcp-docker", "description": "...", "installed": False, "source": "mcp_registry",
   "repository_url": "https://github.com/owner/mcp-docker"},
]
```

**Note on install_cmd for MCP servers:** MCP server installation varies (npm, pip, binary). The registry does not provide a standard install command. For now: omit `install_cmd`, provide `repository_url` only. User must follow the repo's README. Flag this limitation clearly in output.

**Timeout:** 5s (same as npx). Fail silently — return cached data or empty list.

## What Changes for the User

**Before v0.6.0:**
Dispatch shows you 3-6 tools — all things you already have. Haiku picks from your installed collection. Discovery is limited to agent skills via skills.sh.

**After v0.6.0:**
Dispatch shows you the best tools for your task from the entire ecosystem — installed or not. Switching to a Docker task might surface an MCP server you've never heard of, rated 91/100, with a link to install it. Switching to a workflow task might surface a marketplace plugin that's clearly better than what you have. You make the call. Haiku just does the work of knowing what exists.

## Known Limitations (honest)

- **Plugin install_cmd format:** VERIFIED — `claude plugin install {name}@{marketplace} && claude`
- **MCP install_cmd:** Not standardized. Each MCP server has its own install process. We can only provide the repository URL.
- **Smithery:** Not included. Requires user API key. Can be added as opt-in in a future version (store token in `config.json`).
- **Marketplace catalog staleness:** Catalogs were downloaded when the user added the marketplace. They're not auto-updated. Refresh requires re-adding the marketplace or adding a TTL-based re-fetch.
- **MCP registry search is substring-only:** `registry.modelcontextprotocol.io` warns that `search` is a simple substring match. May miss semantically relevant servers with different naming.
- **Token budget:** Sending all sources to Haiku for a broad task type (e.g., "python") could hit max_tokens=600. May need to cap uninstalled categories at 5 each before sending.
